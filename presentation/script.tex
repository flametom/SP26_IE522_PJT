\slide{2}{Recap: Simulating Emergency Evacuation}
\begin{itemize}
\item Brief recap from our 1st presentation. The paper asks: when a hazard hits a community, how do thousands of pedestrians actually evacuate?
\item Each person decides independently on local information, panic causes irrational behavior, congestion is emergent --- a traditional process-flow model cannot capture this, so we need an agent-based simulation.
\end{itemize}

\slide{3}{Recap: Three-Component Model}
\begin{itemize}
\item Three coupled components from the paper: community network, human agents, hazard agents.
\item Two algorithms tie them together --- Algorithm 1 (human--network) and Algorithm 2 (human--hazard). Both run every step.
\end{itemize}

\slide{4}{Recap: Agent States \& Algorithms 1--2}
\begin{itemize}
\item Six states per human, with Algorithm 2 handling hazard exposure and Algorithm 1 handling routing.
\item Two points worth emphasizing: once impacted, destination flips to the nearest shelter; panic is irreversible.
\item We take both algorithms directly from the paper --- no changes to the decision logic.
\end{itemize}

\slide{5}{What we reproduced \& extended}
\begin{itemize}
\item Three reproduction targets from the paper: Phase 1 (five community networks), Phase 2a ($RI$ vs $P_{\max} \times H_{\max}$), Phase 2b ($RS/RC/RL$ vs panic).
\item Three extensions beyond the paper's range: wider $P_{\max}$, wider $H_{\max}$, finer panic grid.
\item The notation box at top defines every symbol used later.
\end{itemize}

\slide{6}{Why speed matters}
\begin{itemize}
\item Our full experiment is about 390 runs. A straightforward per-agent shortest-path implementation in Python takes 5--15 minutes per run, which is days of wall time for the whole sweep.
\item Our pipeline runs the same model in about 32 seconds per run, a few hours total with parallel workers. Critically: we did NOT change the algorithms --- only the functions used for path computation.
\end{itemize}

\slide{7}{Engineering 1: Profiling}
\begin{itemize}
\item The bar chart shows where the time goes. Algorithm 1 path recomputation dominates --- over 80\% of runtime.
\item Root cause: every non-panicked agent triggers its own shortest-path query from its current position, every step, in pure Python.
\end{itemize}

\slide{8}{Engineering 2: Shared shortest-path trees}
\begin{itemize}
\item Key insight: many agents share destinations --- the same building, the same shelter. So compute one shortest-path tree per unique destination; each agent then looks up its route in constant time.
\item This converts $O(N)$ calls per step into $O(K)$ with $K \ll N$. Measured: about 230x speedup on the path-computation section alone, 7.5x end-to-end single-process.
\end{itemize}

\slide{9}{Engineering 3: Per-step recomputation \& panic noise}
\begin{itemize}
\item Per-step recomputation is kept, following the paper --- pedestrians act on up-to-date observations, so paths are not cached across steps.
\item A few edge cases stay per-agent: shelter redirects, congestion-aware detours, panicked agents with noisy edge weights. These are only a few percent of calls.
\item The bottom callout is the key message: Algorithms 1 and 2 are unchanged; only the path function was swapped.
\end{itemize}

\slide{10}{Engineering 4: Parallelism + RNG streams}
\begin{itemize}
\item Eight parallel workers across the seed--config grid give an extra 8x throughput. The network is built once and shared to workers via copy-on-write.
\item Separate RNG streams for humans, hazards, and sim dynamics. Without this, changing $P_{\max}$ would also change hazard placements --- we couldn't tell whether a higher $RI$ came from more people or from different hazards.
\end{itemize}

\slide{11}{Engineering 5: Measured speedup}
\begin{itemize}
\item Head-to-head benchmark at the paper's baseline config: 168 seconds per run with per-agent queries, 22 seconds with shared trees. That is 7.5x single-process, about 60x overall with 8 workers.
\item What this speedup enabled: 10 seeds per config, all 5 communities for Phase 1, three full extensions --- without cutting corners.
\end{itemize}

\slide{12}{PSU-UP baseline in action}

\slide{15}{Phase 2a: $RI$ vs $P_{\max} \times H_{\max}$}
\begin{itemize}

\begin{itemize}

\slide{17}{Gaps with the paper}
\begin{itemize}
\item Four categories of remaining gap, all traced to data the paper did not publish: shelter list, casualty formula, hazard seeds, exact graph preprocessing.
\item For the shelter gap: we swept the shelter fraction and found 62\% of buildings best matches $RS$ at panic = 90\%. Details in the sensitivity table.
\end{itemize}

\slide{18}{Three questions we could ask}
\begin{itemize}
\item Three questions the paper's range didn't answer, which our faster pipeline lets us tackle.
\item Paper tested $P_{\max}$ up to 8K, $H_{\max}$ up to 5, panic at 5 points. We push each axis further in the next three slides.
\end{itemize}

\slide{19}{Extension 1: $P_{\max}$ breakpoint}
\begin{itemize}
\item Extended $P_{\max}$ from 2K to 97K across three $H_{\max}$ levels.
\item Independence holds up to 20K, then breaks at 50K by +8.5\%p.
\item Mechanism: 969 buildings at 20-person occupancy is about 20K capacity. At 50K we overflow --- agents queue outdoors and get hit by hazards.
\end{itemize}

\slide{20}{Extension 2: $H_{\max}$ saturation}
\begin{itemize}
\item Extended $H_{\max}$ from 5 to 30 at paper-baseline $P_{\max}$.
\item $RI$ saturates at about 86\% --- more hazards do not push past this ceiling.
\item Interpretation: likely about 14\% of the network is structurally unreachable by hazards.
\end{itemize}

\slide{21}{Extension 3: Finer panic grid}
\begin{itemize}
\item 9 panic points vs the paper's 5.
\item Curves are smooth, monotonic, with diminishing marginal effect --- no phase transitions between the paper's measured points.
\item Policy implication: panic management has highest payoff at low baseline panic.
\end{itemize}

\slide{25}{Summary}
\begin{itemize}
\item Reproduction successful within 2--4\%p of the paper across all three phases.
\item Three novel findings beyond the paper: $P_{\max}$ breakpoint at 50K, $H_{\max}$ saturation near 86\%, smooth diminishing panic effect.
\item The engineering enabled all of this without changing any paper algorithm --- the six-metric dashboard summarizes the scale.
\end{itemize}

\slide{26}{Limitations}
\begin{itemize}
\item Where we focused: faithful paper reproduction, an engineering improvement, and three extensions.
\item What we left on the table: no agent-model changes, no casualty-formula rewrite, no multi-community extension of Phase 2. The engineering-first stance was deliberate --- a fast pipeline is the prerequisite for any of the agent-side extensions.
\end{itemize}
